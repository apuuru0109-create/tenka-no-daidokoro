# 面白さレビュー運用

## 目的

機能が動くかだけでなく、「プレイヤーが何を考え、何に迷い、何を気持ちよいと
感じるか」を、実装のたびに点検する。

レビューの基礎には、YouTubeチャンネル「桜井政博のゲーム作るには」で扱われた
ゲーム性、企画、UI、演出、効果、仕事の姿勢などの考え方を用いる。

特にこの作品では、次の原則を重視する。

1. ゲーム性を生む中心はリスクとリターン、つまり駆け引きである
2. ゲーム性だけが面白さのすべてではなく、操作、物語、ごっこ遊び、再現にも価値がある
3. プレイヤーが最も長く行う行為を、そのゲームならではの体験にする
4. 入力に対する結果を、表示、音、動き、変化で明確に返す
5. 説明を読ませる前に、遊びながらルールを理解できるようにする
6. 待ち時間、重複操作、不要な手順を減らし、ゲームテンポを守る
7. 重要なものほど見た目と演出に強弱をつける
8. 作り手の想定ではなく、初見プレイヤーの行動を観察して判断する

## 実装ごとの必須レビュー

### 1. 体験の核

- プレイヤーが今回もっとも長く行う操作は何か
- それは「相場の兆しを読み、資金を張る」体験を強めているか
- 数字の作業だけになっていないか

### 2. リスクとリターン

- プレイヤーは何を得ようとして、何を危険にさらすか
- 安全で常に得な選択肢がないか
- 選択肢ごとに異なる作戦が成立するか
- 結果は運だけでなく、判断の改善につながるか

### 3. 分かりやすさ

- 次にできることがひと目で分かるか
- 数字や専門用語の意味が、判断と結びついているか
- 失敗した理由を後から説明できるか
- 重要な情報が装飾や文章に埋もれていないか

### 4. 手応えとフィードバック

- 操作直後に、何が変わったか分かるか
- 成功、危険、損失の強弱が伝わるか
- 価格だけでなく、米俵、人物、町の様子へ結果が返るか

### 5. テンポ

- 同じ意味の操作や待ち時間が繰り返されていないか
- 一日の中に「読む→決める→結果を見る」の波があるか
- 新しい判断が適度な頻度で発生するか

### 6. ゲーム性以外の面白さ

- 商人になりきるごっこ遊びが成立しているか
- 登場人物や町の行く末を知りたいと思えるか
- 米や市場の本格的な再現そのものに発見があるか
- 見た目、音、言葉遣いが作品固有の魅力を作っているか

### 7. 改善判断

問題を感想だけで終わらせず、次の形式で残す。

| 項目 | 内容 |
|---|---|
| 観察 | 実際に起きた行動・迷い・誤解 |
| 原因仮説 | どの仕様がその結果を生んだか |
| 改善 | 最小限の変更案 |
| 成功条件 | 変更後に観察したい行動 |
| 判定 | 維持／再調整／撤回 |

## レビューのタイミング

- 機能を追加した直後
- プレイ可能な節目に到達したとき
- ユーザーから疑問、混乱、退屈、攻略法が報告されたとき
- 数値調整を行う前後
- マイルストーンを完了するとき

レビュー結果は [FUN_REVIEW_LOG.md](FUN_REVIEW_LOG.md) に追記する。
